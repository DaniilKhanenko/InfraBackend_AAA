# InfraBackend_AAA

1. В качестве фреймворка выбрал FastApi, изначально потому что уже знаком с ним и мне казалось, что для подобных сервисов это стандарт. Также, проанализировав остальные фреймворки из примеров, понял, что Flask - синхронный, устаревает. Aiohttp - требует очень много кода, функционал (в рамках выполняемого ДЗ) схож с FastApi.

2. 
- Latency - время ответа пользователю, P50/95/99 - перцентиль - показывает время быстрее которого выполняется n% запросов. С помощью него можно посмотреть, не будет ли тормозить наш сервис со стороны пользователя при нагрузке.
Т.к. у нас очень (50мб) маленькая модель: P50 < 50 мс, P99 < 150 мс.
Что делать при нарушении: оптимизировать модель, код.
- Throughput (RPS) - сколько запросов в секунду может обработать сервис(сервер), с помощью него можно понять сколько пользователей выдержит наш сервис(один докер контейнер).
Наверное 30 RPS будет оптимально для нашей маленькой модельки.
При нарушении нужно проверить правильность использования асинхронности.
- Model inference time - все понятно из названия, позволит понять кто виноват в возможной малой скорости работы сервиса - фреймворк/сеть или все таки модель, в целом можно посмотреть в целях проверки.
Ожидаемый порог: 20-30мс(очень маленькая модель)
Что делать при нарушении: оптимизировать модель
- CPU / Memory usage - сколько памяти и cpu уходит на контейнер. Позволит проконтролировать утечки памяти (например, что мы загружаем модель 1 раз а не несколько на каждый запрос).
Ожидаемый порог - в простое 0, возможны скачки до 100 по cpu но в среднем < 50%
RAM до 1гб
Если есть лик памяти - смотрим код и фиксим.
